# 技术写回和 QA 规则

本文规定审核通过后的资源写回、格式处理和补丁质检。语义审核见 `REVIEW_RULES.md`。

## 1. 进入条件

1. 目标 CN 已完成语义审核，或明确列出尚未解决的问题。
2. 已确认目标章节、资源格式和写回工具。
3. 修改前能够定位：
   - `chapter`
   - `csv_row`
   - `id / stable_id`
   - `resource_file`
   - `scene`
   - `speaker_jp`
   - `jp_text`
   - `cn_text`
4. 未确认记录用途、文本槽或资源格式时，先只读分析，不写回。

## 2. 通用硬规则

1. 不修改 JP 原文和稳定定位字段。
2. 不使用英文槽、旧中文或模糊匹配定位或兜底。
3. 必须使用精确 ID、稳定 ID、文件偏移或已验证的唯一定位条件。
4. 原始游戏文件不得被覆盖；输出到新目录或新文件。
5. 不直接分发完整商业游戏资源，优先制作差分补丁。
6. 不自动生成或公开提交临时 repack、package stage、测试副本和完整资源镜像。
7. 不运行 `git add -A`，不 reset 或 checkout 丢弃其他工作区改动。

## 3. 控制符原则

JP 源字段和 CN 目标字段必须分开判断：

1. JP 源文本、原始控制序列和资源结构不得被破坏。
2. CN 中需要保留哪些控制符，取决于经过验证的资源格式和游戏显示行为。
3. 不采用“所有 JP 控制符一律复制到 CN”的通用规则。
4. 引号、书名号、括号等功能性符号应保留其语义作用；不能无故漏掉外层 `「」`，也不能把 `『』` 随意换成不同功能的符号。

## 4. EGPACK 文本

1. CN 替换文本不得包含字面量 `\n`、字面量 `\r` 或真实 CR/LF 换行。
2. 中文正文依靠引擎自动换行；人为加入换行标记可能造成错误断行和大块空白。
3. `\p` 是消息结束类结构标记，`\f` 可用于控制记录；是否保留必须依据记录结构和已验证工具规则。
4. `\w` 是等待类控制，不是换行。是否保留应依据该游戏和目标文本槽的实机验证结果。
5. `<01>`、`<03>`、ruby 或显示标记按具体记录结构处理，不能凭外观删除。
6. 写回使用精确 `id + slot + expected_text` 或工具规定的等价强校验条件。
7. 输出新 egpack 后重新解析，确认记录数、ID、槽位和非目标字节没有异常变化。

## 5. RIO / CRsa 文本

1. 使用 `stable_id`、RIO 文件、CRsa 块和 payload offset 精确定位。
2. 按已验证算法在内存中解密 payload，以 UTF-16LE 处理目标文本。
3. 保留对应记录所需的二进制控制序列、结束符和槽位边界。
4. 替换后重新加密并写入新 RIO，不覆盖原文件。
5. 重新提取写回结果，与目标 CSV 和原始非目标记录逐项比较。
6. 最终优先制作 byte patch，不公开分发完整 RIO。

## 6. FPD、UI、图片、视频和字体

1. 记录原始资源路径、容器层级、文件格式、尺寸和编码。
2. UI 文本与正文文本分开分类，不能因为都在同一容器中就混用规则。
3. 图片替换保持游戏需要的尺寸、透明通道、色彩和压缩格式。
4. 视频替换检查时长、分辨率、帧率、音轨和编码兼容性。
5. 字体替换必须确认字体授权允许项目当前的公开分发方式，并检查缺字、字宽、基线、描边和实际显示。
6. 非正文资产完成后应进行菜单、选项、存档、回放、结尾字幕等实机路径检查。

## 7. 每次修改后的自动审计

至少检查：

- Text ID Not Found 或定位失败。
- 空 `cn_text` 和意外空记录。
- 行顺序、ID、槽位或偏移错位。
- 异常重复文本。
- 乱码、替换字符和编码损坏。
- 不应残留的日文假名或英文整句。
- BETA、CP、ID 等应保留缩写是否丢失。
- 原文功能性符号是否被错误替换或丢失。
- 控制符是否符合目标资源格式，而不是简单比较字符数量。
- 人名、军衔、称谓和术语一致性。
- 非目标记录或非目标字节是否被意外修改。

## 8. 实机和补丁验证

1. 在原版文件副本上安装补丁，不在开发输出目录直接覆盖唯一原件。
2. 验证启动、主菜单、设置、存档读取、章节进入和目标 scene。
3. 检查自动换行、文本框溢出、等待、翻页、选项和跳转行为。
4. 检查 UI、图片、视频、字体和片尾字幕等非正文资源。
5. 卸载或还原流程必须可用。
6. 补丁包只包含必要差分、安装脚本、说明和授权允许分发的资源。

## 9. QA 留档

每次写回或打包至少记录：

1. 输入文件哈希和工具版本。
2. 目标章节、记录数量和定位方式。
3. 写回成功、跳过、失败和未找到数量。
4. 自动审计结果。
5. 实机验证范围和结果。
6. 已知问题及是否阻止发布。

只有语义审核、自动审计和必要实机验证均完成后，才能把对应资源标记为可发布。
