# 通用翻译和审校规则

本文适用于 TDA00-03、photonflowers、photonmelodies 以及后续章节的文本审校、反馈处理和补丁源表维护。章节对话可以在此基础上补充本章专属角色语气、术语和技术细节，但不能降低本文的硬规则。

## 0. 总原则

1. JP 原文是唯一翻译依据。
2. 不使用英文槽兜底。
3. 不使用旧中文兜底。
4. 不使用模糊匹配兜底。
5. 旧 CN 只能作为待审候选，不能默认可信。
6. 每一句都必须回到 JP 原文、scene、speaker、上下文重新判断。
7. 中文目标是正式汉化文本：自然、通顺、像中国玩家会读到的中文。
8. 不机械保留日文句壳，但必须对 JP 原文负责。
9. 信息不能丢：人名、称谓、术语、固定说法、剧情信息、情绪转折、角色语气都不能随便省。
10. 任何拿不准的术语、称谓、剧情含义，标记为 question，不硬猜。

## 1. 工作边界

1. 只处理当前明确指定的章节、分支或工作区。
2. 不跨章节直接修改字幕正文；跨章节问题先记录并转交对应章节对话。
3. 文本审校阶段不默认写回资源、不生成补丁、不安装到游戏。
4. 只有用户明确要求“写回”“打包”“生成补丁”“安装测试”时，才进入补丁阶段。
5. 文本阶段优先输出候选译文、修改清单、决策记录、问题列表、审计结果和摘要。
6. 不自动生成 repack、package_stage、patch zip、完整 RIO / egpack 副本。
7. 不运行 `git add -A`。
8. 不 reset，不 checkout 丢弃用户或其他对话改动。

## 2. 每批开始前必须先读懂

每批不能直接逐句改。必须先做前置理解：

1. 篇章理解：
   - 判断当前文本属于哪个章节、route、scene 或自然剧情段。
   - 先确认本段剧情在整篇里的位置和作用。
   - 如果章节结构不明确，先记录 question，不硬定。

2. scene 梗概：
   - 这一场发生在哪里。
   - 谁在场。
   - 正在发生什么。
   - 冲突点是什么。
   - 情绪如何变化。
   - 这场如何承接上一场、引出下一场。

3. 角色关系：
   - 说话人和被称呼者是什么关系。
   - 是上级/下级、同学、同僚、敌人、熟人、调侃、尊敬、威胁、试探还是亲近。
   - 称谓必须服从人物关系，不能只看单句。

4. 角色语气：
   - 按本章节既有台词、JP 原文和场景表现建立语气基准。
   - 同一角色在军令、私下、调侃、恐惧、愤怒、旁白等场景中可以有不同文体。
   - 不要让所有角色变成同一种中文腔调。

5. 术语预检查：
   - 先扫出本批人名、军衔、组织、部队、机体、武器、外号、称号、作战名、固定梗、反复短语、カタカナ词、尊称。
   - 与术语表和前文译法对齐。
   - 拿不准先进 question list。

## 3. 逐句审校逻辑

每一行必须给出 keep / revise / question。

1. keep 条件：
   - JP 原意准确。
   - 中文自然。
   - 人物语气正确。
   - 术语、称谓、控制符都没问题。
   - 没有日式残留、翻译腔、旧译名问题。

2. revise 条件：
   - 有误译、漏译、语气不对。
   - 中文不通顺、拗口、文绉绉。
   - 日式残留明显。
   - 术语不统一。
   - 称谓机械翻译。
   - 固定表达被随便本地化掉。
   - 英文/拉丁缩写被错误删除。
   - 原文普通口语被翻成过度书面语。

3. question 条件：
   - JP 可多解，必须依赖后文确认。
   - 术语或称谓无法确定。
   - 可能是梗、固定表达、设定词。
   - 外语句是否保留/翻译不确定。
   - scene 归属或说话人关系不稳。

## 4. 日文准确性规则

1. 逐句确认主语、宾语、否定、反问、条件、因果、转折、时间关系。
2. 日文省略主语时，可以按上下文补出中文主语，但不能乱补剧情。
3. 长定语、被动句、倒装、重复语气可以改成自然中文。
4. 核心信息、情绪、逻辑不能丢。
5. 一句 JP 有多种理解时，看 scene、人物关系、当时情绪、剧情目的，不只看字面。
6. 普通日常对白要自然口语化。
7. 誓词、军令、贵族式说话、宗教/军事语境、中二台词，可以保留对应文风。
8. 原文完整外语句、德文句、英文句原则上先保留原文；如需翻译，另列候选，不直接替换。

## 5. 中文表达规则

1. 中文优先读起来顺。
2. 忠实原意，但不保留日文句壳。
3. 可以拆句、换语序、补必要主语、调整语气词。
4. 不能乱编剧情、乱加设定、乱改人物关系。
5. 普通对白避免“方才”“此即”“恢复清明”“至今为止”这类不自然表达。
6. 除非原文就是古风、誓词、军令、宗教腔、中二腔，否则不要文绉绉。
7. 旁白要清楚、顺滑，不要堆抽象名词。
8. 角色台词要像这个角色会说的话，不要所有人一个腔调。
9. 不能为了“自然”把有命名感、反复出现、有设定意义的词磨没。
10. 一句话总结：不按字面翻，但按原文负责；中文要好读，但不能把关键词、角色味和设定味磨没。

## 6. 英文 / 拉丁字母规则

1. 原文直接出现的英文、拉丁字母、缩写通常保留。
2. 例如 BETA、CP、ID、VIP、OK、DMZ、m/cm、Alternative 4、Alternative 5 不得随便中文化掉。
3. 不能为了中文顺口把 BETA、CP、ID 这类原文专有缩写弄没。
4. 原文完整外语句先保留原文。
5. 如果外语句影响理解，可以另列中文注释候选，但不直接替换原句。
6. 如果英文/拉丁词在语境中明显是称号、武器名、组织名、作战名，可以提出意译候选，并记录理由。
7. 是否最终意译，要写入术语表或等待确认。

## 7. Katakana / 外来语规则

1. カタカナ不默认音译。
2. 常见英文、含义明确、意译更自然或更有气势的词，优先考虑意译。
3. 称号、武器、组织、作战名、英文风格カタカナ，如果意译更清楚、更自然，优先意译。
4. 例如 Black Knife 这类可以优先考虑“黑刃”，而不是机械音译。
5. 人名、品牌名、机体固有名、官方固定写法优先固定译名。
6. 意译会造成误解时，才考虑音译、半音译或保留英文。
7. 重要カタカナ首次决定译法时，必须进入术语表并写理由。

## 8. 术语和固定表达规则

1. 人名、军衔、组织、部队、机体、武器、外号、称号、作战名、固定梗、反复短语必须统一。
2. 同一个专有名词不能一处一个译法。
3. 除非语境确实不同，否则不能随意变体。
4. 有命名感、反复出现、有梗、有设定意义的表达不能为了顺口丢掉。
5. 像“伝家の宝刀”这种固定表达，不能顺手改成普通说法导致识别度消失。
6. 术语优先查已有中文惯用译名。
7. 没有惯用译名，再按语境决定音译、意译或保留英文。
8. 拿不准的术语保守处理，标 question，不硬定。

## 9. 称谓规则

1. さん、ちゃん、様、殿、君不能机械翻。
2. 要按说话人、被称呼者、关系、身份差、场景语气判断。
3. タケルさん不能随便翻成“武先生”，更可能按关系处理成“武同学”等。
4. “xx桑”不是默认答案。
5. “同学”“大人”“阁下”“少尉殿”“小姐”“先生”等都要按关系固定。
6. 如果日式称谓在中文里突兀，要改成符合人物关系的中文。
7. 如果原文刻意使用敬称制造距离感、身份感或滑稽感，要保留这种功能。

## 10. 符号、控制符和技术规则

1. `jp_text` 原文不能破坏。
2. 所有控制符必须保留。
3. 控制符数量、顺序、位置逻辑必须一致。
4. 不随意删除 `\w`、`\n`、`<01>`、`<03>` 等控制符或显示标记。
5. 原文中的引号、书名号、括号等功能性符号原则上保留对应形态；例如 `「」`、`『』` 不应随意改成 `“”`、`‘’`、`【】`。
6. 修改前必须能定位：
   - chapter
   - csv_row
   - stable_id / id
   - rio_file / egpack / resource_file
   - scene
   - speaker_jp
   - jp_text
   - cn_text
7. 不允许出现 Text ID Not Found 风险。
8. 不允许错位写入。
9. 不允许重复异常。
10. 不允许空文本异常。
11. 不允许乱码、替换字符、异常方块源自文本本身。

## 11. 每批输出和留档

每批必须留档，方便回退和检查：

1. candidate：当前批次完整候选译文。
2. overlay：csv_row、stable_id / id、scene、speaker_jp、jp_text、旧 CN、新 CN、修改理由。
3. decision ledger：每行 keep / revise / question，不能批量默认 keep。
4. question list：所有拿不准的术语、称谓、剧情理解、特殊表达。
5. context review：story / scene brief、character notes、term precheck。
6. audit：控制符、空文本、错位、重复、乱码、英文/拉丁字母丢失、BETA/CP/ID 丢失、日文假名残留、Text ID Not Found。
7. summary：本批范围、总行数、keep / revise / question 数量、主要问题、下一批从哪里继续。

这些留档可以放在章节工作区或本地内部目录；公开仓库只保留稳定源表、工具、术语表和通用规范。

## 12. 审计硬要求

每批结束必须检查：

1. Text ID Not Found 风险。
2. 空文本。
3. 台词错位。
4. 重复异常。
5. 乱码/替换字符。
6. 英文/拉丁字母残留或丢失是否符合规则。
7. BETA、CP、ID 等原文缩写是否保留。
8. 控制符数量和顺序是否一致。
9. 原文符号是否被错误替换或丢失。
10. 是否残留不该出现的日文假名。
11. 是否有旧译名残留，例如错误 Luna 写法。
12. 是否有繁体残留。
13. 是否有日式称谓残留。
14. 是否有中文读起来不像中文的问题。
15. 是否有术语前后不统一。
16. 是否有角色语气跑偏。

## 13. 分批推进规则

1. 按 chapter / route / scene / 剧情顺序推进。
2. 不随机抽行。
3. 每批按完整 scene 或自然剧情段切分。
4. scene 太长可以拆，但必须保持上下文连续。
5. 每批先理解，再逐句审校，再审计。
6. 不要半批停在中间。
7. 时间或 token 不够时，必须在批次边界收尾。
8. 每批结束后记录 resume_state 或等价交接信息。
9. 下一次继续时必须知道从哪个 csv_row、scene、stable_id / id 开始。

## 14. 阶段完成标准

一个章节或自然阶段完成时，应满足：

1. 所有已有中文都逐句审校。
2. 所有未翻译行都补译或明确列入 question。
3. 每行都有 keep / revise / question 记录或等价审计结果。
4. 每批都有 overlay、ledger、question list、audit、summary 或等价留档。
5. 所有术语、称谓、固定表达都有可追踪记录。
6. 所有审计问题为 0，或者明确列入 question。
7. 除非用户明确要求，不写回资源，不生成补丁，不安装到游戏。

## 15. 最终判断标准

一句译文合格，必须同时满足：

1. JP 原意准确。
2. 中文自然好读。
3. 角色语气正确。
4. scene 情绪顺。
5. 术语统一。
6. 称谓符合关系。
7. 英文/拉丁/外语处理符合规则。
8. 控制符完整。
9. 原文符号保留合理。
10. 没有日式残留。
11. 放进上下文里不突兀。
